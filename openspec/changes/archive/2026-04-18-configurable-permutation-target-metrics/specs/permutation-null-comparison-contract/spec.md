# Permutation Null-Comparison Contract Specification

## Purpose

- Define the canonical permutation/null-comparison diagnostic for the current MA workflow, including configurable target-metric selection and integer-safe persisted summary typing.

## MODIFIED Requirements

### Requirement: the permutation diagnostic is owned by a dedicated diagnostic module

`src/alphaforge/permutation.py` SHALL be the single authoritative owner of the permutation/null-comparison diagnostic for the current MA workflow.

#### Purpose

- Keep the anti-overfitting check separate from search, validation evidence, and policy verdicts.
- Prevent CLI, runner, or report code from inventing null-comparison semantics locally.

#### Canonical owner

- `src/alphaforge/permutation.py` is the authoritative owner of permutation target-metric selection, permutation evaluation, and null-comparison statistics.
- `src/alphaforge.schemas.py` is the authoritative owner of the in-memory permutation summary contract.
- `src/alphaforge.storage.py` is the authoritative owner of persisted permutation-summary schema and type normalization.
- `src/alphaforge.cli.py` is the authoritative owner of target-metric request parsing.

#### Allowed responsibilities

- The diagnostic module MAY:
  - load market data,
  - evaluate one fixed MA candidate on the original data,
  - evaluate the same fixed candidate on permuted data,
  - compare one selected target metric at a time,
  - compute an empirical p-value from the observed metric value and permutation metric values,
  - return a compact runtime summary for storage and CLI serialization.

#### Explicit non-responsibilities

- The diagnostic module MUST NOT generate candidates.
- The diagnostic module MUST NOT re-run search or re-rank candidates per permutation.
- The diagnostic module MUST NOT emit policy verdicts.
- The diagnostic module MUST NOT own report layout or visualization semantics.

#### Scenario: the diagnostic reuses a fixed candidate

- GIVEN a strategy spec for the MA crossover family
- WHEN the permutation diagnostic runs
- THEN the candidate parameters SHALL remain fixed across all permutations
- AND the diagnostic SHALL compare the real observed metric value against a null distribution produced from permuted data

### Requirement: permutation semantics preserve local temporal structure via contiguous block shuffling

The null distribution SHALL be generated by splitting the ordered OHLCV frame into contiguous non-overlapping blocks, preserving row order inside each block, and shuffling block order only while keeping candidate parameters and backtest configuration fixed.

#### Purpose

- Make the null model easy to explain and test.
- Avoid introducing a more complex statistical framework than the current repo needs.

#### Permutation contract

- the permutation unit is a contiguous block of OHLCV rows
- blocks SHALL be formed from the ordered frame in row order
- rows within each block SHALL keep their original order
- block order SHALL be shuffled only
- a trailing partial block is allowed and SHALL remain as its own block
- the candidate SHALL be evaluated on each permuted frame without reselecting parameters
- the block size SHALL be a positive integer
- the block size SHALL not exceed the number of rows in the loaded data
- the seed SHALL be explicit so the same inputs produce the same null distribution

#### Target metric

- supported target metrics SHALL include `score` and `sharpe_ratio`
- the default target metric SHALL be `score`
- the target metric name SHALL be recorded explicitly in the diagnostic summary

#### Scenario: fixed seed yields the same block null distribution

- GIVEN the same market data, candidate parameters, block size, permutation count, and seed
- WHEN the permutation diagnostic runs twice
- THEN the permutation metric values SHALL be identical in both runs
- AND the empirical p-value SHALL also be identical

### Requirement: the diagnostic summary is compact, stable, and storage-backed

The permutation diagnostic SHALL produce a stable summary artifact with explicit comparison fields and storage-owned artifact references.

#### Purpose

- Keep the output auditable without making the diagnostic a new research workflow engine.
- Ensure CLI output and persisted files stay aligned.

#### Summary fields

- strategy name
- strategy parameters
- target metric name
- permutation mode
- block size
- real observed metric value
- permutation metric values
- permutation count
- seed
- null ge count
- empirical p-value
- artifact paths when produced

#### Persisted artifacts

- the summary artifact SHALL be persisted as JSON
- the metric-value list SHALL be persisted as a tabular artifact
- artifact paths SHALL be owned by storage and surfaced by CLI only as derived refs

#### Comparison statistic

- `null_ge_count` SHALL count the number of permutation metric values greater than or equal to the real observed metric value
- `empirical_p_value` SHALL be calculated as `(null_ge_count + 1) / (permutation_count + 1)`
- integer-like summary fields (`block_size`, `permutation_count`, `seed`, `null_ge_count`) SHALL remain integers in persisted JSON

#### Scenario: summary fields stay stable

- GIVEN a permutation diagnostic run completes
- WHEN the summary is serialized
- THEN it SHALL include the required comparison fields
- AND the empirical p-value SHALL be derived from the recorded permutation metric values
