# permutation-null-comparison-contract Specification

## Purpose
Define the canonical minimal permutation/null-comparison diagnostic for the current MA workflow, including how the null distribution is generated, how a selected target metric is compared against it, and how the persisted summary keeps count-like fields integer-typed.
## Requirements
### Requirement: the permutation diagnostic is owned by a dedicated diagnostic module

`src/alphaforge/permutation.py` SHALL be the single authoritative owner of the permutation/null-comparison diagnostic for the current MA workflow.

#### Purpose

- Keep the anti-overfitting check separate from search, validation evidence, and policy verdicts.
- Prevent CLI, runner, or report code from inventing null-comparison semantics locally.

#### Canonical owner

- `src/alphaforge/permutation.py` is the authoritative owner of the null-comparison computation and target-metric selection.
- `src/alphaforge/schemas.py` is the authoritative owner of the runtime summary and receipt contracts for the diagnostic.
- `src/alphaforge/storage.py` is the authoritative owner of persisted filenames and artifact layout for the diagnostic outputs.
- `src/alphaforge/cli.py` is the authoritative owner of command parsing and terminal output formatting for the diagnostic command.

#### Allowed responsibilities

- The diagnostic module MAY:
  - load market data,
  - evaluate one fixed MA candidate on the original data,
  - evaluate the same fixed candidate on permuted data,
  - compare one selected target metric at a time,
  - compute an empirical p-value from the observed metric value and permutation metric values,
  - return a compact runtime summary for storage and CLI serialization.

#### Explicit non-responsibilities

- The diagnostic module MUST NOT generate candidates.
- The diagnostic module MUST NOT re-run search or re-rank candidates per permutation.
- The diagnostic module MUST NOT emit policy verdicts.
- The diagnostic module MUST NOT own report layout or visualization semantics.

#### Scenario: the diagnostic reuses a fixed candidate

- GIVEN a strategy spec for the MA crossover family
- WHEN the permutation diagnostic runs
- THEN the candidate parameters SHALL remain fixed across all permutations
- AND the diagnostic SHALL compare the real observed score against a null distribution produced from permuted data

### Requirement: permutation semantics preserve local temporal structure via contiguous block shuffling

The null distribution SHALL be generated by converting ordered OHLCV rows into relative OHLC movement rows, splitting those relative rows into contiguous non-overlapping blocks, shuffling block order, and reconstructing a continuous synthetic price path from the original starting price while keeping candidate parameters and backtest configuration fixed.

#### Null model

- the canonical null model SHALL be named `return_block_reconstruction`
- the first market-data row SHALL be kept as the synthetic path anchor
- for each row after the first row, relative movement SHALL be computed against the previous close:
  - `open_rel = open_t / close_{t-1}`
  - `high_rel = high_t / close_{t-1}`
  - `low_rel = low_t / close_{t-1}`
  - `close_rel = close_t / close_{t-1}`
- contiguous blocks SHALL be formed from these relative rows
- rows within each relative block SHALL preserve their original order
- block order SHALL be shuffled only
- volume values MAY be permuted with the same relative rows
- synthetic OHLC values SHALL be reconstructed from the previous synthetic close:
  - `synthetic_open = previous_synthetic_close * open_rel`
  - `synthetic_high = previous_synthetic_close * high_rel`
  - `synthetic_low = previous_synthetic_close * low_rel`
  - `synthetic_close = previous_synthetic_close * close_rel`
- reconstructed OHLC values SHALL be finite and positive
- reconstructed `high` SHALL be at least the maximum of reconstructed open, high, low, and close
- reconstructed `low` SHALL be at most the minimum of reconstructed open, high, low, and close

#### Permutation contract

- the permutation unit is a contiguous block of relative OHLC movement rows
- blocks SHALL be formed from the ordered relative rows in row order
- a trailing partial block is allowed and SHALL remain as its own block
- row count SHALL be preserved
- canonical OHLCV columns SHALL be preserved
- datetime ordering shape SHALL be preserved from the original market data
- the candidate SHALL be evaluated on each reconstructed synthetic frame without reselecting parameters
- the block size SHALL be a positive integer
- the block size SHALL not exceed the number of loaded market-data rows
- the seed SHALL be explicit so the same inputs produce the same null distribution

#### Scenario: return-block reconstruction avoids absolute price stitching

- GIVEN market data containing two flat absolute price regimes at materially different price levels
- WHEN relative movement rows are block-permuted and reconstructed from the original anchor price
- THEN the synthetic path SHALL remain continuous from the previous synthetic close
- AND it SHALL NOT introduce a jump solely because an absolute high-price block followed a low-price block

### Requirement: the diagnostic summary is compact, stable, and storage-backed

The permutation diagnostic SHALL produce a stable summary artifact with explicit comparison fields, null-model metadata, and storage-owned artifact references.

#### Summary fields

- strategy name
- strategy parameters
- target metric name
- null model
- permutation mode
- block size
- real observed metric value
- permutation metric values
- permutation count
- seed
- null ge count
- empirical p-value
- artifact paths when produced

#### Scenario: summary identifies null construction method

- GIVEN a permutation diagnostic run completes
- WHEN the runtime summary is returned or serialized
- THEN it SHALL include `null_model`
- AND `null_model` SHALL equal `return_block_reconstruction`
- AND `block_size`, `permutation_count`, and `seed` SHALL remain explicit in the summary

### Requirement: the permutation diagnostic supports every search-supported fixed candidate family it can construct explicitly

`src/alphaforge/permutation.py` SHALL build a fixed diagnostic candidate for both `ma_crossover` and `breakout`, returning the base `Strategy` type while keeping candidate parameters fixed across all permutations.

#### Purpose

- Let the diagnostic operate on the repo's supported fixed-candidate families without turning the permutation workflow into a search or registry system.
- Keep diagnostic strategy construction owned by `permutation.py` while the supported family list remains owned by `search.py`.

#### Canonical owner

- `src/alphaforge/permutation.py` remains the authoritative owner of permutation diagnostic strategy construction.
- `src/alphaforge/search.py` remains the authoritative owner of the supported strategy-family set.
- `src/alphaforge/strategy/base.py` remains the authoritative owner of the shared `Strategy` interface.

#### Allowed responsibilities

- `permutation.py` MAY construct either `MovingAverageCrossoverStrategy` or `BreakoutStrategy` from a fixed `StrategySpec`.
- `_build_strategy()` MAY return the shared `Strategy` base type.
- the diagnostic MAY reject only strategy families that are outside the supported family set or that it cannot construct explicitly.

#### Explicit non-responsibilities

- `permutation.py` MUST NOT generate candidates or rerun search per permutation.
- `permutation.py` MUST NOT define a concrete return type that falsely narrows the supported family set to MA crossover only.
- `cli.py` MUST NOT assemble a hardcoded MA-only `StrategySpec` for `permutation-test`.

#### Inputs / outputs / contracts

- Inputs:
  - `DataSpec`
  - fixed `StrategySpec`
  - permutation count, block size, target metric, seed, and backtest config
- Output:
  - a diagnostic summary for the fixed candidate
- Contract rules:
  - the fixed candidate parameters remain unchanged across permutations
  - supported target metrics remain `score` and `sharpe_ratio`
  - supported fixed candidate families include `ma_crossover` and `breakout`

#### Invariants

- One fixed candidate is evaluated on the real data and each permuted frame.
- Diagnostic family support stays aligned with the canonical search-supported family names.

#### Scenario: breakout uses the same fixed-candidate diagnostic contract

- GIVEN a breakout `StrategySpec`
- WHEN the permutation diagnostic runs
- THEN the diagnostic SHALL build one fixed breakout strategy instance per evaluation
- AND it SHALL compare the selected target metric against block-permuted null samples without rerunning search
